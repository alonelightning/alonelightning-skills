# Social Media Content Workflow

多平台社交媒体内容创作全链路——从选题到发布。

**核心理念：约束 > 创意。** Skill 的价值在于定义"什么时候做、怎么做、做到什么标准"。

> **内容产出原则（硬约束）：不设每日篇数指标，质量优先，有多少好内容就发多少。** 不为凑数牺牲质量（宁缺毋滥），也不因“今天已发够”而放过好内容；评估选题时给出完整“可做清单”+优先级，由决策者定多少。

## 工作流总览（12 步）

```
1. 任务理解       → 协调者
2. 触发 Skill     → 自动判断
3. 环境确认       → 代理检查 ⚠️ 阻断点
4. 文件夹创建     → 脚本/手动 ⭐ 一开工就在长期归档位建当日文件夹
5. 资料收集       → 执行者 + 搜索工具
5.5 素材归档      → 记录来源 URL ⚠️ 强制
6. 文案+配图关键词 → 内容创作者 ⭐ 一步完成
7. 审核提意见     → 审核者
8. 综合判断       → 协调者
9. 改稿+改关键词  → 内容创作者 或 协调者（小改动）
9.5 终审确认      → 协调者/用户 ⚠️ 阻断点
10. 关键词检查    → 协调者 ⚠️ 阻断点
11. 出图         → Codex CLI ⚠️ 阻断点
12. 交付         → 用户手动发布
```

> ⭐ **步骤 6 是文案与配图关键词一步完成。** 内容创作者写稿时必须同时输出完整的四段式 Prompt。

## 协作模式：选题对齐 → 执行全权（硬约束）

适用于多角色/多 Agent 协作：**选题阶段一次性对清，之后执行层直接推进、不中途打断。**

- **选题阶段（步骤 1-2）**：把三件事一次性问透定定——问题（要不要做/风险）、方向（核心立意）、内容（看点/覆盖哪些平台）。理解偏了代价大，拿不准就直接问。
- **执行阶段（步骤 3-12）**：选题定定后，文案措辞/字数/emoji/标签/配图细节/出图校验返工全部直接定、不回来问；全流程跑完后一次性交验收。
- **例外（仍需回来问）**：红线新分歧、需改变已定方向、改稿超 2 轮仍不达标。

## 建议模型配置

| 环节 | 推荐模型 | 备选 |
|------|---------|------|
| 信息收集 | DeepSeek V4 Pro | Claude Opus 4（成本更高） |
| 选题策划/协调 | Claude Opus 4 | GPT-5 系列 |
| 审核 | GPT-5 系列 | 建议与协调者用不同模型 |
| 文案+配图关键词 | Claude Opus 4 | — |
| 出图执行 | Codex CLI -m gpt-5.5 | — |

> **关键设计**：审核者与内容创作者使用不同模型，形成多角度交叉验证。
> **多篇并行**：每篇一个独立内容创作者会话，避免单人串行赶多篇（防多篇互相污染致幻，见下文）。

## 平台规范速查

| 平台 | 风格 | 正文字数 | 配图尺寸 |
|------|------|---------|---------|
| 小红书 | 短文案，数据入图 | ≤200字（硬上限300） | 9:16（1080×1920） |
| 视频号 | 极简，核心信息 | ≤100字（硬上限150） | 9:16（1080×1920） |
| 微信公众号 | 深度分析 | 3000-5000字 | 封面 2.35:1 |

### 正文内容分配原则
- **正文只放**：钩子 + 情绪/感受 + 互动引导（提问/CTA）+ 标签
- **配图承载**：所有硬信息（数据、对比、参数、定价、步骤、列表）
- AI 出图可直接渲染中文文字信息，放心把信息塞进图里
- 不硬编文案凑字数；信息少时正文可以只有 2-3 句话

## 约束清单

### 硬性约束

| 约束 | 说明 |
|------|------|
| 出图必须走 Codex CLI | `codex exec -m gpt-5.5 -s danger-full-access --skip-git-repo-check` |
| Prompt 写入文件 | 先写 `.md`（UTF-8），再让 codex 读取；禁止命令行传中文 |
| 多路并行出图 | ≤5 路并行，禁止单张串行 |
| Prompt 四段式格式 | 见下方详细模板 |
| 代理检查强制 | 出图前必须确认代理状态 |
| 改稿 ≤2 轮 | 超过请求用户介入 |
| 正文字数硬上限 | 小红书 ≤200（不超300）；视频号 ≤100（不超150） |
| 信息入图 | 硬信息必须做进配图 |
| 文件名英文 | Codex 不支持中文文件名 |

### 质量标准

| 标准 | 说明 |
|------|------|
| 事实可追溯 | 所有数据必须能追溯到素材来源 |
| 不确定用语气词 | 未证实信息使用"据报道"、"据反映" |
| 配图质量检查 | >1MB = AI 生成；<500KB = 重做 |
| 文本渲染检查 | 出图后用视觉模型校验图上文字正确性 |

### 红线禁忌

- ❌ 不编造数据、引语、事件细节
- ❌ 不夸大事实
- ❌ 不用营销号体（"彻底爆了"、"看完沉默了"等）
- ❌ 不擅自切换出图路径
- ❌ 不跳过代理检查和素材归档
- ❌ 敏感内容（政治/宗教/未证实负面新闻）需人工确认

### 🔴 来源归属铁律（最高红线，违反即废稿）

> **只要内容来自别人（博主/作者/演讲者/机构/某篇文章/某条推文），就必须点名到具体的人或出处。不点名 = 据为己有 = 废稿。**

- ❌ 禁「有人分享」「网友分享」等匿名化模糊来源——必须点名：是谁、出自哪里
- ❌ 禁用「我/我们」视角把他人的体验/观点/成果重写成自己的
- ❌ 禁直接搬用他人方法/观点/框架却不提是谁提出的
- ✅ 正确做法：第三人称转述 + 点名（「X 在他的《YY》里提到…」）；事实可直陈，观点/创见/方法必归属到提出者
- **判据**：读者读完能否分清“哪些是自己的、哪些是别人的、是谁的”？分不清 = 不合格

### 🔴 素材铁律：必带可核验链接，禁凭记忆填来源（编造级红线）

- ✅ 每一条素材必附可核验原文链接（URL）或账号原始 ID；博主名/作者名/专有名词/数据均需从链接核实，**不允许凭记忆填写**
- ✅ 拿不准的专有名词（尤其公众号/博主名）宁可标“待核实”，不允许猜一个填上
- ❌ 禁把“印象中的名字”当事实写入素材（编造来源）；发现即作废重拉

> “幻觉式来源”（凭印象编一个看着像模像样的博主名/出处）比压缩丢信息更危险——看着可信其实全错。宁标“待核实”，绝不猜填。

## 配图 Prompt 四段式标准结构（⭐ 核心）

### 结构（必须按此顺序）

```
[第1段] 要什么图：用处 + 尺寸 + 类型 + 主要风格
[第2段] 背景：一段完整英文插画背景描述，引号圈起来
[第3段] 背景与文案的关系：overlay 或 blend seamlessly
[第4段] 文案信息：固定框架 + 文案排版位置，文案用引号圈起来
```

### 完整模板

```
I need to create an image for posting on Xiaohongshu, with a 9:16 aspect ratio and dimensions of 1080x1920 pixels: This is a [卡片类型], with a [风格描述] style. Visual style: Illustrated background; [区域色调描述]. The keywords for the background are "[60-120词完整英文插画背景描述, 末尾加 no text, plenty of space for overlay]." [关系句]. The image must include the following text (please ensure the text is clear and legible): [按位置标注的中文文案，每条用双引号]
```

### 两种策略

| 卡片类型 | 策略 | 关系句 |
|----------|------|--------|
| 金句/概念/情绪 | A 融合型 | `Finally, I need to overlay some text on the background, and I want the text to blend seamlessly with the illustrated background.` |
| 步骤/框架/要点/数据 | B 结构型 | `Finally, I need to overlay some text on the background.` |

### 风格倒向

| 内容类型 | 风格 |
|----------|------|
| 产品/信息发布（新技术、新模型、产品更新） | 深色调科技风 |
| 知识/内容分享（偏应用层、面向大众） | 插画风、清新轻盈 |
| 硬核技术知识（偏底层） | 深色调信息卡 + 插图装饰 |

### 必须检查的要点
1. 尺寸：9:16 / 1080×1920
2. 背景描述：60-120 英文词，引号圈起
3. 图上文案：必须中文，双引号框起
4. 第3段关系句：不可省略
5. 第4段固定开头：`The image must include the following text (please ensure the text is clear and legible):`
6. 署名：底部带创作者署名

## 出图执行规范

### 执行流程

1. **写 prompt 文件**：将每张 prompt 写入 `prompt-<name>.md`（UTF-8）
   - 末尾加：`Generate image with size 1024x1536 and save as <name>.png`

2. **codex 读文件生图**：
   ```bash
   codex exec -m gpt-5.5 -s danger-full-access --skip-git-repo-check -C "<产出目录>" "读取当前目录下的 prompt-<name>.md 文件，用其内容作为 prompt 调用 image_gen 工具生成图片，保存为 <name>.png"
   ```

3. **并行**：一次 ≤5 路 background，跑完一批再发下一批

4. **轮询**：一批发完后一次长等待，不高频 poll

5. **判定**：>1MB = 成功；<500KB = 重做；自动重试 1 次

6. **文本渲染检查**：出图全部完成后，用视觉模型逐张检查图上文字正确性
   - 错字/漏字/排版严重错乱 = 重做
   - 轻微变形但可识别 = 标记警告，由协调者决定

7. **收尾**：删除 prompt-*.md 中间文件

### 禁止
- ❌ 命令行直接写中文 prompt
- ❌ stdinData 传中文/emoji
- ❌ PowerShell 包装脚本
- ❌ HTML/CSS/SVG/Pillow 代替 AI 生图
- ❌ 降级 codex 模型
- ❌ 工作目录指向含敏感文件的位置

## 协作流程建议

### 多篇并行：单篇单会话防干扰致幻（硬性）
- 多篇同类任务，每篇分配独立内容创作者会话，一篇一个，不交叉
- **根因**：多篇共用一个会话会把多篇原文塞进同一上下文，不同文章的人名/术语/观点互相污染，导致归属错位、细节串位等幻觉。即使同题材也不合并写，宁可多开几个
- 调研全部完成后统一派写稿，出图 ≤5 路并行，由独立执行者会话完成

### 改稿职责

| 改动类型 | 谁来改 |
|----------|--------|
| 小改动（改一两个词、调事实、微调描述） | 协调者直接改 |
| 大改动（重写段落、大幅调配图方案） | 退回内容创作者 |

### 审核链路
```
内容创作者写稿(步骤6) → 审核者审核(步骤7) → 协调者综合判断(步骤8) → 改稿(步骤9)
```

## 错误处理

| 错误 | 处理 |
|------|------|
| `Reconnecting... 5/5` | 等待 30 秒后重试 |
| `failed to refresh available models: timeout` | 忽略，继续 |
| `rmcp transport worker quit` | 直接重试 |
| 图片 <500KB | 重新生成 |
| 代理未开启 | 阻断流程 |
| 改稿超 2 轮 | 请求用户介入 |
