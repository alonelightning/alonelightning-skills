## 一、理念篇：为什么用 Agent 工作流做内容？

### 内容创作的痛点

在没有 Agent 工作流之前，一个人做多平台内容推广是什么体验？让我们还原完整的手动工作流——这就是我们要自动化掉的原始流程。

#### 手动工作流：人肉信息快递员

**Step 1 — 信息收集（30-60 分钟）**
工具：DeepSeek 网页端 / Google 搜索

打开 DeepSeek 网页端，让它直接搜索热点事件的相关信息并整理汇总。但 DeepSeek 的输出不能直接采信——你需要多轮沟通来筛选信源、调整搜索方向。同时手动用 Google 确认关键信息的真实性，交叉验证后才能作为素材。

😩 **痛点**：虽然 AI 帮你做了大部分搜索和整理工作，但你仍然需要全程盯着屏幕——多轮对话筛选信源、手动用 Google 交叉验证、把最终确认的信息整理到本地文档。你是"人肉信息质检员"。

**Step 2 — 选题策划（15-30 分钟）**
工具：Claude 网页端 / Claude Code / ChatGPT

把收集好的信息粘贴给 AI（Claude 网页端、Claude Code 或 ChatGPT 都可以），让它帮忙分析选题角度、确定各平台的内容策略。讨论几轮后，把最终的选题方案手动整理到文档。

😩 **痛点**：无论用哪个 AI 工具，它的上下文都是独立的——看不到你在 DeepSeek 里收集信息的过程，你必须手动把所有信息"搬运"给它。

**Step 3 — 文案创作（60-100 分钟）**
工具：Claude 网页端 / Claude Code / ChatGPT

需要逐个平台产出文案。不一定每个平台单独开一次新对话——你可能有固定的对话对应不同平台，也可能开一次新对话处理所有平台。但每个不同的文案产出都需要重新调整关键词约束和风格要求。Claude Code 可以并行处理不同平台的文案输出——只要提前每个平台开好一次对话、做好约束调整。

😩 **痛点**：把选题方案、参考资料、平台规范重新粘贴给每个对话是真实的重复劳动；对着电脑等待每篇文案的生成过程，这段时间你什么都做不了。

**Step 4 — 配图生成（约 30 分钟）**
工具：ChatGPT 网页端（直接输入 prompt 生成图片）

切换到 ChatGPT 网页端，把配图 prompt 粘贴进去，等它生成图片。一张一张来——输入 prompt、等待生成、检查质量、不满意就重新生成、满意了手动下载保存。每张图大约需要 60-100 秒的生成时间。

😩 **痛点**：ChatGPT 网页端一次只能生成一张图，无法并行。你必须一直盯着，手动触发下一张。

**Step 5 — 审核发布（30-60 分钟）**
工具：手动逐平台操作

把文案和配图分别上传到各个平台：打开微信公众号后台、排版、上传图片、预览、发布；打开小红书 App、逐帖编辑、上传图片、发布；打开博客后台、粘贴文章……

😩 **痛点**：每个平台的操作完全独立，没有任何自动化可言。

#### 手动工作流总耗时

```
1. 信息搜索整理:     30-60 分钟     ← DeepSeek 网页端 + Google 验证
2. 选题讨论:          15-30 分钟     ← Claude / ChatGPT
3. 多平台文案:        60-100 分钟    ← Claude / Claude Code / ChatGPT
4. 配图生成(16张):    约 30 分钟     ← ChatGPT 网页端
5. 审核发布:          30-60 分钟     ← 手动逐平台
─────────────────────────────────
总计:                2.5-4.5 小时
```

**核心问题不是某一步慢，而是整个流程你必须全程在线。** 虽然大部分实际工作已经由各 AI 平台完成，但你始终在做"人肉信息快递员"——在不同 AI 工具之间搬运上下文、等待生成、手动触发下一步。2.5-4.5 小时里你被钉在电脑前，无法去做其他工作或休息。

### Agent 工作流的核心理念

**不是用 AI "帮你写"，而是让多个 AI Agent 组成团队，像人类团队一样分工协作。**

```
手动：1个人 × 2.5-4.5小时 × 全程在线 × 你什么都做不了
Agent：多个 Agent × 并行执行 × 流程化 × 1-2小时 × 可以去做其他事
```

关键思维转变：

1. **分工而非替代**：每个 Agent 有明确的角色和专长，就像人类团队的编辑、设计师、运营分工
2. **流程化而非临时起意**：选题→文案→配图→发布，每步有标准输入输出
3. **约束而非自由发挥**：给 Agent 明确的规范（字数、风格、格式），比"帮我写篇文章"有效 100 倍
4. **复用而非重造**：沉淀为 Skill（技能包），下次直接调用
5. **并行而非串行**：Agent 之间可以同时工作，配图生成可以批量并行，不再需要人肉盯着

### 预期收益

| 指标 | 手动方式 | Agent 工作流 |
|------|----------|-------------|
| 单次推广耗时 | 2.5-4.5 小时（全程在线） | 1-2 小时（大部分是等待，可离开做其他事） |
| 多平台覆盖 | 通常只发1-2个（精力不够） | 3个平台全覆盖 |
| 配图质量 | ChatGPT 网页端逐张生成 | Codex CLI 批量并行生成 |
| 信息传递 | 手动复制粘贴 | Agent 间自动流转 |
| 可复制性 | 低（依赖个人经验） | 高（Skill 标准化） |

## 二、架构篇：工作流全景与角色分工

### 工作流架构图

```
用户灵感/热点事件
  → ① 信息收集（协调者 Agent 或辅助决策 Agent）
  → ② 选题策划 & 平台分配（协调者 Agent）
  → ②b 辅助决策（辅助决策 Agent — 信息验证 + 选题辩论）
  → ③ 内容创作（内容创作者 Agent — 文案 + 配图方案）
  → ④ 配图关键词输出（内容创作者 Agent — 视觉化 prompt）
  → ④b 关键词审查（协调者/用户 — 对照检查清单）
  → ⑤ 配图生成（Codex CLI + GPT 聊天模型）
  → ⑥ 审核 & 发布
```

> ⚠️ **不要跳过第④步**。配图方案（结构描述）≠ 配图关键词（视觉化 prompt）。直接用配图方案生成图片会导致 PPT 级别输出。

### 角色定义

#### 协调者 Agent（Coordinator）
- **能力**：信息搜索、选题策划、任务分配、Codex CLI 调用
- **推荐模型**：Claude Opus 4 系列或同级推理模型
- **职责**：统筹全流程，是工作流的核心调度者

#### 辅助决策 Agent（Advisor）（可选）
- **能力**：信息收集与真实性验证、多角度分析、观点辩论
- **推荐模型**：GPT-5.5（与协调者使用不同模型，形成多角度辩论）
- **职责**：
  - 辅助信息收集，对关键事实进行交叉验证
  - 在选题策划阶段提供与协调者不同的观点，通过辩论完善方案
  - 辅助选题决策，避免单一模型的偏见盲区
- **说明**：这不是多设备协作的概念，而是在同一个工作环境中引入第二个视角。就像团队里有两位编辑——一位负责统筹（协调者），另一位负责质疑和补充（辅助决策者），通过观点碰撞产出更高质量的决策。

#### 内容创作者 Agent（Content Creator）
- **能力**：多平台文案创作、配图方案设计
- **职责**：根据平台规范产出文案，每篇附带标准化配图 prompt

#### 用户（最终决策者）
- **职责**：审核文案和配图、最终发布、效果复盘

### 模型建议

不同环节对模型能力的要求不同，以下是基于实战经验的推荐搭配：

| 环节 | 推荐模型 | 理由 |
|------|---------|------|
| 信息收集 | DeepSeek V4 Pro | 高性价比，搜索和信息整理能力强，token 成本低 |
| 选题策划 | Claude Opus 4 系列 | 逻辑严密，擅长结构化分析和多角度思考 |
| 辅助决策/辩论 | GPT-5.5 | 与 Claude 形成互补视角，适合提供不同观点 |
| 文案创作 | Claude Opus 4 系列 | 文字质量高，风格把控能力强 |
| 配图生成 | Codex CLI -m gpt-5.5 | Codex CLI 聊天模型内部自动调用 image_gen |

> **备注**：Claude Opus 4 系列和 GPT-5 系列亦可用于信息收集，但 token 成本更高。如果预算有限，可全部使用 DeepSeek V4 Pro。

### 平台规范速查

| 平台 | 风格 | 字数 | 配图 | 核心特征 |
|------|------|------|------|----------|
| **微信公众号** | 深度分析 | 3000-5000字 | ~8张 | 编辑署名，数据驱动，观点鲜明 |
| **小红书** | 短平快 | 150-200字/帖 | 4张/帖 | emoji丰富，视觉驱动，互动收尾 |
| **博客** | 散文感 | 不限 | 按需 | 第一人称，情感共鸣，星空意象 |

## 三、流程路径：13 步标准流程

以下为本工作流从触发到交付的完整标准流程。每一步标注了执行主体（LLM / 脚本 / 用户），⚠️ 标记处为阻断点——必须通过当前步骤确认才能继续。

```
1. 任务理解 → LLM
2. 触发 Skill → LLM 判断
3. 环境确认 → 脚本 ⚠️ 阻断点
4. 文件夹创建 → 脚本
5. 资料收集 → LLM + 搜索
5.5 素材归档 → 记录来源 ⚠️ 强制
6. 文案撰写 → LLM
7. 审核提意见 → LLM
8. 综合判断 → LLM
9. 改稿 → LLM
9.5 终审确认 → LLM/用户 ⚠️ 阻断点
10. 出图关键词 → LLM
11. 出图 → 脚本 ⚠️ 阻断点
12. 文件整理 → 脚本
13. 交付 → 用户手动
```

### 步骤详解

| 步骤 | 名称 | 执行主体 | 输入 | 输出 | 阻断点 |
|------|------|---------|------|------|--------|
| 1 | 任务理解 | LLM | 用户原始请求 | 任务摘要 + 平台判断 | 否 |
| 2 | 触发 Skill | LLM 判断 | 任务摘要 | 加载 social-media-content Skill | 否 |
| 3 | 环境确认 | 脚本 | — | 代理状态报告 | ⚠️ 是 |
| 4 | 文件夹创建 | 脚本 | 平台 + 日期 | 任务目录结构 | 否 |
| 5 | 资料收集 | LLM + 搜索 | 任务主题 | 结构化素材文档 | 否 |
| 5.5 | 素材归档 | 记录来源 | 素材文档 | 带来源标注的素材 | ⚠️ 强制 |
| 6 | 文案撰写 | LLM | 素材 + 平台规范 | 各平台文案初稿 | 否 |
| 7 | 审核提意见 | LLM | 文案初稿 | 审核意见列表 | 否 |
| 8 | 综合判断 | LLM | 审核意见 | 综合改稿指令 | 否 |
| 9 | 改稿 | LLM | 综合指令 | 修正后文案 | 否 |
| 9.5 | 终审确认 | LLM/用户 | 最终文案 | 通过/再改 | ⚠️ 是 |
| 10 | 出图关键词 | LLM | 配图方案 | 视觉化 prompt | 否 |
| 11 | 出图 | 脚本 | prompt 列表 | 图片文件 | ⚠️ 是 |
| 12 | 文件整理 | 脚本 | 任务目录 | 整理后目录 + 清单 | 否 |
| 13 | 交付 | 用户手动 | 整理后内容 | 各平台发布 | 否 |

### 阻断点说明

- **步骤 3（环境确认）**：代理不通 → 所有步骤无法进行，必须先修复。
- **步骤 5.5（素材归档）**：不记录来源 → 后续无法追溯事实，违规。
- **步骤 9.5（终审确认）**：用户未确认 → 不得启动出图。
- **步骤 11（出图）**：Codex 不可用或配额耗尽 → 图片产出无法完成。

## 四、约束清单：三层约束体系

本工作流运行过程中，所有 Agent 和脚本必须遵守以下三层约束。违反任一层约束均视为不合格产出。

### 第一层：路径约束（硬性规则）

路径约束是工作流运行的轨道，脱离轨道即视为流程失败。

| # | 规则 | 说明 |
|---|------|------|
| 1 | **出图必须走 Codex CLI** | 不得通过任何其他路径（如 DALL·E API、Midjourney 等）生成配图 |
| 2 | **Prompt 格式固定** | 必须使用附录 A 的标准格式，不得自由发挥 |
| 3 | **代理检查强制** | 每次任务开始前必须执行 `check-proxy.ps1`，代理不通不开始 |
| 4 | **改稿轮次上限 2 轮** | 超过 2 轮改稿仍未通过 → 标记任务并向上汇报 |
| 5 | **出图尺寸规范** | 小红书 1:1 · 公众号 2.35:1 · 微博 16:9 · 严格按平台标准 |
| 6 | **单次任务超时 30 分钟** | 从步骤 1 到步骤 13 总时长不超过 30 分钟（不含用户等待时间） |
| 7 | **文件名必须英文** | 所有脚本产出、图片、中间文件的文件名统一使用英文，中文仅保留在文件内容中 |

### 第二层：质量标准（产出底线）

质量标准是可检查的产出底线，不达标即为不合格。

| # | 规则 | 检查方式 |
|---|------|---------|
| 1 | **事实可追溯** | 所有数据和引用必须有来源链接或文档出处 |
| 2 | **不确定用语气词** | 无法确认的信息必须使用「据报道」「据称」「可能」等措辞 |
| 3 | **融资/财务谨慎** | 涉及金额、融资、收入等数据时，加倍验证后仍需注释来源 |
| 4 | **信息量匹配内容量** | 1000 字文案必须有足够的信息密度，不注水不凑数 |
| 5 | **配图质量检查** | 文件 >1MB = AI 生成合格 · 文件 <500KB = 疑似程序化绘制，需重做 |

### 第三层：禁忌（红线）

禁忌是不可触碰的红线，触碰即视为事故。

| # | 禁忌 | 说明 |
|---|------|------|
| 1 | **不编造数据** | 不得虚构任何数字、日期、引语、事件 |
| 2 | **不夸大事实** | 不得使用「史上最强」「颠覆行业」「唯一」等绝对化描述（除非有权威认证） |
| 3 | **不用营销号体** | 不得使用「震惊！」「竟然！」「一定要看！」等标题党句式 |
| 4 | **不擅自切换出图路径** | 即使 Codex 报错也不得切换到其他图片工具 |
| 5 | **不跳过代理检查** | 不得以「上次通过了」为由跳过环境确认 |
| 6 | **不跳过素材归档** | 不得省略步骤 5.5 中的来源记录 |
| 7 | **敏感内容需人工确认** | 涉及政治、社会争议、敏感人物时，必须经用户确认后才能发布 |

### 🔴 两条最高红线（违反即废稿）

**来源归属铁律**：只要内容来自别人（博主/作者/演讲者/机构/某文/某推文），就必须点名到具体的人或出处。禁「有人分享」「网友分享」等匿名化模糊来源，禁用「我/我们」把他人成果重写成自己的。判据：读者能否分清“哪些是自己的、哪些是别人的、是谁的”。

**素材铁律**：每条素材必带可核验原文链接（URL）；博主名/专有名词/数据需从链接核实，禁凭记忆填写。拿不准宁可标“待核实”，不允许猜一个填上（“幻觉式来源”看着可信其实全错，比丢信息更危险）。

> 配合机制：内容创作者**写稿前必先点开原文链接复核**，不只凭压缩素材写；多篇并行时**一篇一个独立创作者会话**，不合并（防多篇原文在同一上下文互相污染致幻）。

## 五、脚本使用说明

工作流中部分步骤由自动化脚本执行，所有脚本位于你的工作区脚本目录（示例：`<workspace>/scripts/social-media/`），可按自己的项目结构存放。

### 脚本清单

| 脚本 | 用途 | 示例 |
|------|------|------|
| `check-proxy.ps1` | 检查代理状态 | `cd <脚本目录>; .\check-proxy.ps1` |
| `init-folder.ps1` | 创建任务目录 | `.\init-folder.ps1 -Platform xiaohongshu -Date 2026-05-23` |
| `gen-image.ps1` | 调用 Codex 生成图片 | `.\gen-image.ps1 -Prompt "..." -OutputPath "xxx.png"` |
| `organize-files.ps1` | 整理文件并生成清单 | `.\organize-files.ps1 -TaskDir "<任务目录>"` |

### 脚本说明

**check-proxy.ps1** — 阻断点脚本
- 测试系统代理是否连通 OpenAI API
- 输出：通过（继续）/ 失败（阻断，需修复后重试）
- 每次任务必须执行，不得跳过

**init-folder.ps1** — 任务初始化
- 参数：`-Platform`（xiaohongshu / wechat / weibo / blog），`-Date`（YYYY-MM-DD）
- 产出：`{平台}-{日期}/` 目录结构，含子目录 `文案/` `配图/` `素材/`

**gen-image.ps1** — 配图生成
- 参数：`-Prompt`（视觉化 prompt），`-OutputPath`（输出文件路径）
- 内部调用 `codex exec -m gpt-5.5`

**organize-files.ps1** — 交付前整理
- 参数：`-TaskDir`（任务目录路径）
- 产出：重命名文件为规范命名 + 生成 `交付清单.md` 列出所有文件

## 六、配图 Prompt 检查清单

> 每次生成配图 prompt 后，对照本清单逐项检查。缺任一要素即为不合格 prompt。

### 7 个必须包含的要素

| # | 要素 | 要求 | 示例 |
|---|------|------|------|
| 1 | **背景渐变+方向** | 必须写清楚渐变颜色和方向 | `dark navy to black vertical gradient` |
| 2 | **视觉效果层** | 至少包含 1 种特效 | `particle effects`, `glassmorphism`, `HUD elements` |

### 检查通过标准

- [ ] 7 要素全部包含
- [ ] 词数在 120-200 之间
- [ ] 不包含中文（引号内文案除外）
- [ ] 开头含 `Use the image generation tool to create this image` 防护语

## 八、踩坑篇：所有已知问题与解决方案

> ⚠️ 这一章是整篇文档最重要的部分。所有踩坑都来自实战，每一条都可能让你卡住几十分钟。

### 坑1：gpt-image-2 不能作为 -m 参数

```
❌ codex exec -m gpt-image-2 "prompt"
→ 连接失败 + Reconnecting...
```

**原因**：gpt-image-2 是纯图片生成模型，不是聊天模型。Codex CLI 的 `-m` 参数只接受聊天模型（能对话的模型）。

**正解**：
```powershell
✅ codex exec -m gpt-5.5 "prompt"
```

使用 gpt-5.5（或其他 GPT 聊天模型），模型会**内部自动调用** image_gen 工具来生成图片。

**为什么会犯这个错**：直觉上觉得"生成图片就应该用图片模型"，但 Codex CLI 的工作方式是：你和一个聊天模型对话，聊天模型自己决定调用哪些工具（包括图片生成工具）。

**铁律**：**永远不要把 gpt-image-2 作为 `-m` 参数。**

### 坑2：failed to refresh available models: timeout

```
ERROR codex_models_manager::manager: failed to refresh available models: 
timeout waiting for child process to exit
```

**影响**：这只是 WARNING 级别的提示，**不影响后续执行**。

**处理**：忽略它，继续等待任务完成。不要因为看到这个错误就中断或重启任务。

### 坑3：Reconnecting 错误

```
ERROR: Reconnecting... 1/5
ERROR: Reconnecting... 2/5
...
ERROR: Reconnecting... 5/5
```

**原因**：网络连接问题，**最常见的原因是系统代理没有开启**。

**排查步骤**：
1. 检查系统代理是否开启
2. 用浏览器访问 `api.openai.com` 确认能连通
3. 如果代理已开，可能是代理节点问题，换个节点试试

**预防措施**：每次使用 Codex CLI 前，养成先检查代理的习惯。

### 坑4：rmcp transport 错误

```
ERROR rmcp::transport::worker: worker quit with fatal: Transport channel closed
```

**原因**：网络不稳定导致 WebSocket 连接断开。

**处理**：直接重试命令即可，通常是一次性的网络波动，不需要额外排查。

### 坑5：首次使用需要登录

**现象**：第一次运行 codex exec 提示未认证。

**解决**：
```powershell
codex login
```

浏览器会弹出 OpenAI 登录页，登录后凭证自动保存。

**OpenClaw / EasyClaw 特殊情况**：内置的 Codex CLI 路径比较深：
```
%LOCALAPPDATA%\easyclaw\ai\tool_cache\resources\tools\win\node-{版本}\codex.cmd
```

如果你在 PowerShell 里手动调用，需要使用完整路径，或把这个目录加到系统 PATH 环境变量。

### 坑6：图片保存位置不对

**现象**：prompt 里说了"保存到当前目录"，但图片找不到。

**原因**：Codex 的默认行为是先把图片存在 `~/.codex/generated_images/` 下，然后自己执行 copy 到目标路径。如果 prompt 中没有明确的保存指令，图片可能只存在缓存目录。

**正解**：prompt 必须包含明确的保存路径和文件名：
```
...Generate and save as cover.png in the current directory.
```

**最佳实践**：先 `cd` 到目标目录，再执行 codex exec，prompt 中指定 "in the current directory"。

### 坑7：并行任务轮询过快

**现象**：同时启动多个 codex exec 后频繁检查状态，导致系统负担加重。

**正解**：轮询间隔 ≥ 10-15 秒。单张图片生成通常需要 30-60 秒。

```
推荐并行策略：
- 同时启动 4 个进程
- 每 15 秒检查一次状态
- 完成一个就启动下一个
```

### 踩坑速查表

| 症状 | 原因 | 解决 |
|------|------|------|
| Reconnecting 5/5 | 代理没开 | 开启系统代理 |
| gpt-image-2 连接失败 | 用了图片模型作 -m 参数 | 改用 gpt-5.5 |
| timeout refresh models | 正常 WARNING | 忽略 |
| transport channel closed | 网络波动 | 重试 |
| 图片找不到 | prompt 没指定路径 | 加 "save as X.png" |
| 未认证 | 没登录 | codex login |

## 九、进阶篇：自动化与 Agent 间协作

### 10.1 信息收集的前置流程

用户的想法是：不在选题时才临时搜索信息，而是**常态化信息收集**。

```
日常状态:
  协调者 Agent / 辅助决策 Agent
  → 持续收集目标领域信息
  → 形成信息汇总池
  → 热点事件发生时，从池中提取 + 增量搜索

选题触发:
  用户灵感 → 调取信息池 → 快速形成选题方案
```

辅助决策 Agent 可以在日常信息收集阶段就参与进来——对收集到的信息进行真实性验证和多角度补充。当选题触发时，协调者 Agent 和辅助决策 Agent 通过辩论快速完善方案。

### 10.2 OpenClaw / EasyClaw Agent 间任务分发

在 OpenClaw（或 EasyClaw 等封装版本）中，协调者 Agent 可以将任务分发给子 Agent：

```
协调者 Agent 分发选题和规范给内容创作者 Agent
  → 内容创作者 Agent 在独立子会话中创作文案
  → 完成后自动汇报给协调者 Agent
  → 协调者 Agent 汇总、审核、启动配图生成
```

这种模式的好处：
1. 各 Agent 专注自己的领域
2. 并行执行不互相阻塞
3. 产出有标准格式，便于下游处理

### 10.3 批量配图自动化

当配图数量较多（比如 16 张）时，手动逐个执行不现实。推荐的自动化策略：

```powershell
# 伪代码：批量生成脚本
$posts = @("1a", "1b", "2", "3")
$maxParallel = 4

foreach ($post in $posts) {
    # 为每个帖子创建目录
    mkdir "xhs-images/$post"
    
    # 启动 4 个并行进程生成该帖子的 4 张图
    for ($i = 1; $i -le 4; $i++) {
        codex exec -m gpt-5.5 -s danger-full-access "prompt for pic$i..." &
    }
    
    # 等待完成（轮询间隔 ≥ 15 秒）
}
```

在 OpenClaw / EasyClaw 中，通过 `exec` + `background` 启动后台进程，`process poll` 检查状态。

### 10.4 Skill 驱动的标准化

本工作流已沉淀为 `social-media-content` Skill，包含：
- 平台规范（字数、风格、结构模板）
- 配图 prompt 标准格式
- Codex CLI 调用规范和排错指南
- 团队分工约定

当 Agent 收到"帮我做社交媒体推广"类请求时，会自动加载此 Skill，按标准流程执行。

### 10.5 质量反馈循环

每次推广结束后：
1. 收集各平台数据（阅读量、互动量、转发量）
2. 对比不同帖子的表现
3. 迭代优化文案模板和配图风格
4. 更新 Skill 中的规范

## 附录 C：踩坑速查卡片

| 症状 | 原因 | 一句话解决 |
|------|------|-----------|
| Reconnecting 5/5 | 代理没开 | 开系统代理 |
| gpt-image-2 连接失败 | 模型选错 | 改 gpt-5.5 |
| timeout refresh models | 正常现象 | 忽略 |
| transport channel closed | 网络抖动 | 重试 |
| 图片找不到 | 没指定路径 | prompt 加保存指令 |
| 未认证 | 没登录 | codex login |

#### ❌ 中文文件名 / 中文路径编码问题

**现象**：Codex 遇到中文文件名时反复重试、速度极慢，或者 fallback 到 Pillow 程序化绘制。

**正解**：文件名统一用英文（`cover.png`, `pic1.png`, `data.png`），prompt 中的中文描述保留。

#### ❌ Codex fallback 到 Pillow/PIL 程序化绘制

**现象**：Codex 不调用 image_gen，而是写 Python 代码用 Pillow 画图。产出文件小（<500KB），质量极低，像 PPT。

**正解**：prompt 开头加防护：`Use the image generation tool to create this image, do NOT write code or use Pillow/PIL.`

**检查标准**：>1MB = AI 生成，<500KB = 大概率 Pillow 画的，需重做。

#### ❌ OpenAI API 配额耗尽

**处理**：记录已完成/未完成的图片，设定定时任务在配额重置后自动补齐。批量生成前估算数量，超过 10 张时提前警告配额风险。

#### ⌨️ `--skip-git-repo-check` 参数

当工作目录不在 Git 仓库内时，Codex 会报错退出。加上 `--skip-git-repo-check` 解决。
